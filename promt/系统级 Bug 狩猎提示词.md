# 系统级 Bug 狩猎提示词

> 整理时间：2026-04-26
> 适用工具：Claude Code、Cursor、Copilot、OpenCode + omo、Codex CLI、Windsurf 等所有支持多轮对话的 AI 编程工具

---

## 0. 定位

一份**工具无关**的多轮迭代 Bug 狩猎提示词模板。粘贴到任何 AI 编程工具中，AI 会自动执行多轮穷尽式 Bug 挖掘——不需要你手动反复提问。

---

## 1. 核心原理

绝大多数 AI 工具在单轮对话中只会给出表面级分析。通过在提示词中**强制设定迭代规则**，可以让任何 AI 工具进入多轮深度分析模式：

1. **第一轮**：全景扫描，梳理链路，列出可疑点
2. **第二轮**：深入验证，复现路径，定位根因
3. **第三轮**：边界与并发，极端值、竞态条件
4. **第四轮**：跨系统联调，调用链、数据传递、状态一致性
5. **第 N 轮**：安全与性能，泄漏、阻塞、超时、崩溃
6. **收敛**：连续两轮无新发现时输出最终报告

---

## 2. 输入材料准备

提示词效果直接取决于你提供的材料质量。尽量多给，AI 分析越精准：

| 材料类型 | 优先级 | 示例 |
| --- | --- | --- |
| 核心代码 / SQL | 必须 | 业务逻辑、数据操作、接口实现 |
| 业务流程描述 | 必须 | 用户操作路径、状态流转图 |
| 数据库 schema | 高 | 表结构、索引、约束 |
| 接口定义 | 高 | API 签名、请求/响应格式、错误码 |
| 架构/调用关系 | 按需 | 服务拓扑、消息队列、缓存策略 |
| 已知问题 | 按需 | 偶现的 bug、用户反馈的异常现象 |

⚠️ **材料越多，误报越少、漏报越少。** 只贴一段代码而不给上下文，AI 只能做语法级审查。

## 3. 通用提示词模板（直接复制使用）

```text
你现在是【高级自动化 Bug 狩猎专家】。
我需要你使用【多轮迭代、自动反思、自动深入】的方式，穷尽式查找所有 Bug 和潜在风险。不允许只给一轮结果就停止。

执行规则：
1. 每一轮必须输出：当前轮次编号 + 本轮分析结论 + 发现的 Bug/风险 + 下一步深入方向
2. 自动进行至少 5 轮独立迭代分析，每一轮都要比上一轮更深入
3. 必须覆盖以下维度（每轮聚焦不同维度）：
   - 逻辑错误（条件判断、状态转换、返回值处理）
   - 边界错误（空值、溢出、数组越界、零除）
   - 并发错误（竞态条件、死锁、数据竞争）
   - 数据异常（不一致、丢失、重复、脏数据）
   - 接口错误（参数校验、超时、重试、幂等性）
   - 兼容性问题（版本、编码、时区、平台差异）
   - 性能隐患（N+1 查询、内存泄漏、无索引、全表扫描）
   - 安全漏洞（注入、越权、敏感数据泄露、CSRF/XSS）
4. 涉及跨模块/跨服务时，必须追踪完整调用链路
5. 每轮结束后自我反思：是否有遗漏？是否可以更深入？是否有隐藏问题？
6. 直到连续两轮没有新 Bug 发现，再输出最终汇总报告

现在开始对以下系统进行【多轮穷尽式 Bug 排查】：
--------
【请在这里粘贴你的系统描述、业务流程、接口、代码片段】
--------
```

⚠️ **输出截断续传**：大多数 AI 工具有单次回复长度限制。如果 AI 还没迭代完就被截断了，直接回复"继续"，AI 会从上一轮接着往下分析。

### 3.1 增强技巧

让 AI 对每个发现的 Bug 输出标准化的分析：

```text
额外规则：每发现一个 Bug，必须按以下格式输出：
- 触发条件：什么操作/数据会触发
- 影响范围：影响哪些功能/用户/数据
- 复现步骤：具体的操作序列
- 根因分析：为什么会发生
- 严重程度：P0(数据丢失/安全) / P1(核心功能不可用) / P2(体验受损) / P3(隐患)
- 修复建议：具体的修复方案
```

## 4. 代码级审查模板

如果要查**代码 Bug**（而非系统级），使用这个更聚焦的模板：

```text
使用多轮迭代模式审查以下代码。每轮深入一个维度，至少迭代 5 轮：

第 1 轮：语法与类型 — 类型错误、空指针、未处理异常
第 2 轮：逻辑与状态 — 条件分支遗漏、状态机缺陷、返回值未检查
第 3 轮：并发与线程 — 竞态条件、死锁、可见性问题
第 4 轮：资源与性能 — 内存泄漏、连接泄漏、资源未释放
第 5 轮：安全与边界 — 注入、越权、输入校验、编码问题

每轮必须输出新发现的问题。直到连续两轮无新问题，输出《代码审查汇总报告》。

代码：
--------
【粘贴代码】
--------
```

## 5. 跨系统排查模板

排查涉及多个服务/模块的跨系统 Bug 时使用：

```text
你现在是【跨系统 Bug 狩猎专家】。
对以下系统进行多轮穷尽式排查，必须追踪完整调用链路。

执行规则：
1. 必须追踪每条请求的完整路径：入口 → 服务A → 服务B → 数据库 → 回调/响应 → 返回
2. 任何一环的数据不一致、状态不匹配、时序问题都视为 Bug
3. 重点检查：分布式事务一致性、重试幂等性、消息丢失/重复、超时级联失败
4. 至少 5 轮迭代，每轮聚焦一条调用链路
5. 连续两轮无新发现后输出《跨系统 Bug 汇总报告》

系统架构与交互描述：
--------
【粘贴系统架构图/服务调用关系/接口定义】
--------
```

## 6. 按场景提示词速查

不同开发场景需要不同的排查侧重点。以下模板在通用模板基础上做了场景化裁剪，直接复制使用。

### 6.0 场景选择速查

| 场景 | 推荐模板 | 核心差异 |
| --- | --- | --- |
| 上线前全面排查 | 通用模板（第 3 节） | 全维度穷尽式扫描 |
| 生产环境故障排查 | 6.1 生产故障排查 | 从现象反推根因，追踪日志和链路 |
| Code Review 排查 | 6.2 Code Review 审查 | 逐行审查，关注可维护性和团队规范 |
| 重构/迁移前评估 | 6.3 重构前风险盘点 | 找隐藏依赖和破坏性变更风险 |
| 并发/性能问题 | 6.4 并发与性能专项 | 专注竞态、锁、资源泄漏 |
| 安全审计 | 6.5 安全漏洞专项 | OWASP Top 10 + 业务逻辑越权 |
| 第三方集成排查 | 6.6 外部依赖与集成 | 接口契约、超时重试、版本兼容 |
| 快速排查 | 极简版（第 8 节） | 5 行搞定 |

### 6.1 生产故障排查

适合：线上报错、用户投诉、监控告警、偶发性故障。从现象出发，反推根因。

```text
线上系统出现异常，需要紧急排查。从以下现象出发，反推根因并排查关联风险：

现象描述：
--------
【粘贴错误日志 / 用户反馈 / 监控告警 / 异常现象】
--------

排查规则：
1. 第 1 轮：现象分析 → 从报错信息/日志推断可能原因（至少列出 3 个假设）
2. 第 2 轮：链路追踪 → 从入口到出错点，逐环节排查数据流和状态变化
3. 第 3 轮：根因定位 → 验证最可能的假设，定位到具体代码行/配置/数据
4. 第 4 轮：影响评估 → 受影响的功能范围、数据范围、用户范围
5. 第 5 轮：关联风险 → 相同模式是否存在于其他模块？是否有级联故障风险？
6. 第 6 轮：修复验证 → 提出修复方案，分析修复是否可能引入新问题

每轮输出：当前假设 + 验证结论 + 下一步方向
最终输出：《故障分析报告》含根因、影响范围、修复方案、预防措施

相关代码/配置：
--------
【粘贴相关代码、配置文件、数据库查询】
--------
```

### 6.2 Code Review 审查

适合：PR 审查、代码走查、合并前检查。关注代码质量和团队规范。

```text
对以下代码变更进行多轮 Code Review，至少迭代 3 轮：

第 1 轮：正确性 — 逻辑是否正确？边界条件是否处理？返回值是否检查？
第 2 轮：安全性 — 是否有注入、越权、敏感数据暴露？输入校验是否充分？
第 3 轮：可维护性 — 命名是否清晰？是否有重复代码？是否违反团队规范？
第 4 轮（如有必要）：性能 — 是否有不必要的查询/计算？资源是否正确释放？

每个问题按以下格式输出：
- 位置：文件名:行号
- 类别：Bug / 安全 / 性能 / 可维护性 / 规范
- 严重程度：Must Fix / Should Fix / Nitpick
- 描述：问题是什么
- 建议：怎么改

代码变更：
--------
【粘贴 diff 或代码】
--------
```

### 6.3 重构前风险盘点

适合：大规模重构、框架升级、数据库迁移前。提前发现可能踩的坑。

```text
即将对以下系统进行重构/迁移，请先进行风险盘点，找出所有可能导致破坏性变更的隐患：

重构/迁移计划：
--------
【描述重构目标，如"从 REST 迁移到 GraphQL"、"数据库从 MySQL 迁移到 PostgreSQL"】
--------

现有代码/架构：
--------
【粘贴相关代码、接口定义、数据库 schema、调用关系】
--------

排查规则：
1. 第 1 轮：接口兼容性 → 公共 API 签名、返回格式、错误码是否变化？调用方是否会受影响？
2. 第 2 轮：数据兼容性 → 数据结构、序列化格式、数据库 schema 是否变化？迁移是否有数据丢失风险？
3. 第 3 轮：隐藏依赖 → 是否有未显式声明的依赖（反射、动态调用、配置注入、事件监听）？
4. 第 4 轮：运行时行为 → 并发行为、初始化顺序、资源生命周期是否会变化？
5. 第 5 轮：回滚方案 → 如果重构后出问题，能否回滚？数据迁移是否可逆？

最终输出：《重构风险清单》按严重程度排序，每项含影响范围、发生概率、缓解措施
```

### 6.4 并发与性能专项

适合：高并发场景、性能瓶颈、压测前检查。专注竞态条件和资源问题。

```text
对以下系统进行并发与性能专项排查，至少迭代 4 轮：

第 1 轮：竞态条件 → 共享状态的读写是否有保护？check-then-act 是否原子？
第 2 轮：锁与同步 → 锁粒度是否合理？是否有死锁风险？锁持有时间是否过长？
第 3 轮：资源管理 → 连接池、线程池、文件句柄是否正确获取和释放？是否有泄漏？
第 4 轮：热点路径 → 高频调用的代码是否有不必要的同步/IO？缓存策略是否合理？
第 5 轮（如有必要）：极端场景 → 流量突增、慢依赖、部分节点故障时的表现

每个问题输出：触发条件 + 并发场景描述 + 性能影响 + 修复方案

代码/系统描述：
--------
【粘贴代码、架构图、压测配置】
--------
```

### 6.5 安全漏洞专项

适合：安全审计、上线前安全检查、合规要求。专注 OWASP Top 10。

```text
对以下系统进行安全漏洞专项审计，至少迭代 4 轮：

第 1 轮：注入攻击 — SQL 注入、命令注入、XSS、模板注入
第 2 轮：认证与授权 — 越权访问（水平/垂直）、会话管理、Token 安全
第 3 轮：数据安全 — 敏感数据明文存储/传输、日志泄露、错误信息泄露
第 4 轮：业务逻辑 — 金额篡改、条件竞争（先收货后退款等）、接口滥用
第 5 轮（如有必要）：依赖与配置 — 第三方库已知漏洞、默认密码、调试接口未关闭

每个漏洞输出：攻击向量 + 攻击步骤 + 影响评级（CVSS） + 修复方案

代码/接口/配置：
--------
【粘贴代码、API 定义、认证配置、中间件配置】
--------
```

### 6.6 外部依赖与集成排查

适合：对接第三方 API、微服务集成、SDK 升级后。关注接口契约和容错。

```text
排查以下系统的外部集成风险，至少迭代 3 轮：

第 1 轮：接口契约 → 请求/响应格式是否与文档一致？字段类型/必填项是否匹配？
第 2 轮：错误处理 → 超时、重试、降级策略是否合理？是否有重试风暴风险？
第 3 轮：数据一致性 → 本地状态与远端状态是否会不一致？回调/通知是否可靠？
第 4 轮（如有必要）：版本兼容 → API 版本升级是否会影响现有调用？SDK 版本是否有 breaking change？

每个问题输出：风险点 + 触发场景 + 影响范围 + 缓解措施

集成描述：
--------
【粘贴接口文档、SDK 版本、调用代码、错误处理逻辑】
--------
```

## 7. 实战示例

### 输入

```text
你现在是【高级自动化 Bug 狩猎专家】。
我需要你使用【多轮迭代、自动反思、自动深入】的方式，穷尽式查找所有 Bug 和潜在风险。不允许只给一轮结果就停止。

执行规则：
1. 每一轮必须输出：当前轮次编号 + 本轮分析结论 + 发现的 Bug/风险 + 下一步深入方向
2. 自动进行至少 5 轮独立迭代分析
3. 必须覆盖：逻辑错误、边界错误、并发错误、数据异常、接口错误、兼容性问题、性能隐患、安全漏洞
4. 跨系统问题必须追踪全调用链路
5. 每轮结束后自我反思
6. 直到连续两轮没有新 Bug 发现，再输出最终汇总报告

现在开始对以下系统进行【多轮穷尽式 Bug 排查】：
--------
用户下单系统：
1. 用户提交订单 → 扣库存 → 生成订单 → 支付 → 完成
2. 库存表：product_id, stock
3. 订单表：order_id, user_id, product_id, status
4. 扣库存SQL：UPDATE product SET stock=stock-1 WHERE product_id=%s
--------
```

### AI 输出（示例）

```
【第 1 轮：全景扫描】
发现 Bug：
1. 无事务保护，扣库存与生成订单可能不一致
2. 无超卖检查，stock 可能变为负数
3. 无并发锁，高并发下会超卖
下一步：进入事务与并发分析

【第 2 轮：事务与并发深入】
发现 Bug：
4. 未加 FOR UPDATE 行锁，并发扣库存会覆盖
5. 订单状态可能出现"已支付但无库存"
下一步：边界值与异常流程

【第 3 轮：边界与异常】
发现 Bug：
6. stock=0 时仍可下单
7. 网络中断会导致半执行状态
下一步：数据一致性

【第 4 轮：数据一致性】
发现 Bug：
8. 无回滚机制，扣库存失败订单仍生成
9. 无幂等性控制，用户重复提交会重复扣库存
下一步：安全与性能

【第 5 轮：安全与性能】
发现 Bug：
10. SQL 直接拼接存在注入风险
11. 无库存预警、无监控告警

【第 6 轮：交叉验证与补充扫描】
反思回顾前 5 轮发现，检查是否有遗漏维度。
未发现新 Bug。本轮为空，连续一轮无新发现。

【第 7 轮：最终确认】
再次审视完整链路，确认无遗漏。
未发现新 Bug。连续两轮无新发现，迭代结束。

======== 最终 Bug 汇总报告 ========
P0: #1 无事务保护 → 数据不一致
P0: #2 无超卖检查 → 库存为负
P0: #3/#4 并发超卖 → 行锁缺失
P1: #8 无回滚机制 → 半执行状态
P1: #9 无幂等性 → 重复扣款
P2: #6 零库存仍可下单
P2: #7 网络中断无容错
P3: #10 SQL 注入风险
P3: #11 缺少监控
```

## 7. 反模式

| ❌ 反模式 | ✅ 正确做法 |
|---|---|
| 只发一句"帮我查 bug" | 使用结构化模板，明确迭代规则和覆盖维度 |
| 手动一轮轮追问 | 在提示词中设定自动迭代规则，让 AI 自行深入 |
| 只关注代码语法 | 每轮聚焦不同维度（并发、安全、性能、数据一致性） |
| 接受单轮结果就结束 | 要求连续两轮无新发现才能输出最终报告 |
| Bug 描述太笼统 | 要求每个 Bug 输出：触发条件、复现步骤、根因、修复建议 |
| 只查代码不查系统 | 根据场景选择对应模板（代码级/系统级/跨系统） |

## 8. 极简版模板（5 行快速版）

适合轻量交互场景（Copilot Chat、Gemini CLI 等），不想贴长提示词时使用：

```text
对以下代码/系统进行多轮 Bug 排查。至少迭代 3 轮，每轮聚焦不同维度（逻辑/边界/并发/数据/安全），
直到无新发现。每个 Bug 给出触发条件、根因、严重程度、修复建议。

【粘贴代码或系统描述】
```

---

> **一句话总结：** 给 AI 一个结构化的多轮迭代规则，比手动反复追问更高效——关键是设定"必须穷尽、必须反思、无新发现才收工"的约束。
